## 可视为真实可用

这些功能不是完美，但已经能在本地 demo 里真实工作。

### 首页提醒中心

状态：可用。

依据：
- 浏览器打开 `/f/demo` 成功。
- 能看到到期统计、位置筛选、食物卡片、消耗/吃完/丢弃按钮。
- 服务端 `performFoodAction` 会更新 `Food` 并写入 `Operation`。

限制：
- 还没有真实权限校验，任何能访问 demo URL 的人都可以操作 demo 数据。

### 位置列表和位置详情

状态：可用。

依据：
- 浏览器打开 `/f/demo/locations` 成功。
- 能看到各个位置、库存数量、最近到期食物。
- 每个位置能进入详情页。

限制：
- 默认种子位置多数没有真实图片，位置卡片仍显示“未拍照”。

### 手动添加食物

依据：

- 表单校验还比较基础。

### 操作记录

状态：可用。

依据：

## 半成品 / 有条件可用

这些功能有页面或代码，但不能再说“已经完成”。

### 语音添加食物

状态：半成品。

已有能力：
- 页面有“开始说话”入口。
- 使用浏览器 `SpeechRecognition` / `webkitSpeechRecognition`。
- 识别结果会进入确认表单。
- 文本解析逻辑有单元测试。

真实限制：
- 依赖浏览器能力，不是所有浏览器都支持。
- 本次没有真实麦克风录音验证。
- 解析只适合比较规整的话术。
- 位置匹配依赖已有位置名称，用户说法稍有变化就可能匹配失败。

结论：
- 只能标为“实验性可用”，不能标为稳定功能。

### 拍日期识别

状态：半成品。

已有能力：
- 页面有上传日期照片入口。
- 服务端会尝试 Gemini、OpenRouter、OpenAI。
- 大模型失败后会回退本地 OCR。
- 识别结果会进入确认表单。

真实限制：
- 大模型识别依赖 API Key、代理和网络。
- 当前 `.env` 未必配置可用 key。
- 本地 OCR 对真实包装图准确率不可保证。
- 本次没有用真实包装照片做端到端识别验证。

结论：
- 只能标为“有识别链路，但真实准确率未验证”。

### 上传柜子图片生成卡通封面

状态：半成品，不能算真正 AI 卡通生成。

已有能力：
- 新增位置页能上传照片。
- 当前实现会在浏览器端用 Canvas 生成封面预览。
- 已从“黑白滤镜”改成“尝试识别木色柜子主体、去背景、色块化、描边”的启发式方案。

真实限制：
- 不是 AI 语义抠图。
- 不是真正的图像生成。
- 对木色柜子可能有效，对白色冰箱、透明箱、复杂光照、深色柜子不稳定。
- 本次没有完成用户提供的“五层柜子.webp”在真实页面上传后的视觉验收。

结论：
- 只能标为“启发式卡通封面实验版”。
- 不能标为“上传照片自动生成好看的卡通柜子”。

### 文件上传

状态：本地 demo 可用，生产不可用。

已有能力：
- `/api/upload` 会把文件保存到 `public/uploads`。

真实限制：
- 没有文件大小限制。
- 没有严格 MIME 校验。
- 没有鉴权。
- 没有云存储。
- 部署到 Vercel 或无持久磁盘环境后不能依赖本地目录。

结论：
- 只能算本地试用方案。

### 页面视觉和配色

状态：可浏览，但仍在试衣阶段。

已有能力：
- 页面整体可用。
- 家庭页有配色切换。

真实限制：
- 配色切换更像开发调试入口，不一定适合最终用户。
- 部分页面信息密度和文字还需要按“妈妈用户”继续打磨。

## 不能视为完成

这些需求目前不能算完成。

### 真实登录和权限

状态：未完成。

原因：
- 当前只有 demo member + cookie。
- 没有正式用户表、账号体系、登录态、会话过期、权限校验。
- 服务端 action 主要靠 familyId/memberId 参数和 cookie，不是安全边界。

### 正式多家庭系统

状态：未完成。

原因：
- 当前邀请码只服务 demo 家庭。
- 没有创建家庭、管理家庭、退出家庭、多家庭切换。

### 生产数据库

状态：未完成。

原因：
- 当前使用 SQLite 本地文件。
- 还没有 Postgres 或其它云数据库配置。

### 线上部署

状态：未完成。

原因：
- 没有完成线上环境部署和验收。
- 上传文件、数据库、环境变量都还依赖本地开发环境。

### 微信小程序

状态：未开始。

原因：
- 当前是网页 MVP。
- 还没有小程序端页面、微信登录、订阅消息。

### 推送提醒

状态：未完成。

原因：
- 目前只有站内提醒中心。
- 没有定时任务、微信通知、邮件、短信或 Web Push。

### 稳定的 E2E 验证

状态：主流程已恢复，覆盖仍需扩展。

原因：
- Playwright 浏览器已补装。
- 当前 MVP 主流程 E2E 已通过。
- 已覆盖真实新增食物提交、库存消耗按钮、记录页库存变化摘要。
- 已覆盖位置图片上传、卡通封面预览、新增位置保存、已上传位置图片更换和封面重新生成。
- 还没有覆盖语音、OCR 等高风险路径。

## 最高优先级修复建议

1. 修复 E2E 环境和测试文案。
   - 安装 Playwright 浏览器。
   - 更新 `tests/e2e/mvp.spec.ts` 为当前真实页面文案。
   - 覆盖：首页 -> 位置 -> 新增位置 -> 手动添加食物 -> 操作 -> 记录 -> 家庭/加入。

2. 把“卡通封面”重新定义清楚。
   - 如果只是本地 Canvas：文案必须写“生成简化封面预览”，不要写 AI 卡通。
   - 如果目标是真正好看的卡通柜子：需要接 AI 抠图/生成服务，并用用户样图验收。

3. 给高风险功能加真实性标签。
   - 语音：实验性。
   - 拍日期：需要 API Key，结果需确认。
   - OCR：兜底，不保证准确。
   - 上传：本地试用，不适合生产。

4. 先修“真实日常闭环”。
   - 手动添加食物。
   - 位置管理。
   - 消耗/吃完/丢弃。
   - 操作记录。
   - 当前使用者。

5. 再做生产化。
   - 登录。
   - 权限。
   - Postgres。
   - 云存储。
   - 部署。

## 当前一句话结论

这个项目现在是“能本地演示和试用的库存 MVP 骨架”，不是“稳定可交付的家庭库存产品”。核心手动库存链路比较真实；语音、拍照识别、卡通封面、权限、部署都还需要按半成品处理。
